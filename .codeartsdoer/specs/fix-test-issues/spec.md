# 需求规格文档

## 文档信息
- **特性名称**: fix-test-issues
- **创建日期**: 2026-05-25
- **版本**: 1.0
- **状态**: 草稿

## 1. 概述

### 1.1 特性描述
本特性旨在修复中文编程语言项目中的关键测试失败问题，确保词法分析器、语法分析器、语义分析器和宏展开系统能够正确处理中文语法元素。修复范围涵盖四个核心模块的测试失败问题，保证所有相关测试用例通过。

### 1.2 业务价值
- **质量保障**: 确保编译器核心组件功能正确，提升系统可靠性
- **用户体验**: 正确处理中文操作符和语法元素，提供流畅的中文编程体验
- **系统稳定性**: 修复语义分析错误检测机制，增强编译时错误提示能力
- **功能完整性**: 完善宏展开功能，支持高级编程范式

### 1.3 范围界定

**包含范围**:
- 词法分析器中文操作符识别修复
- 语法分析器一元操作符和多分号处理修复
- 语义分析器错误检测方法补全
- 宏展开器参数映射和AST节点生成修复

**排除范围**:
- 新增功能特性开发
- 性能优化工作
- 代码重构（除非修复问题必需）
- 其他未失败的测试用例修改

## 2. 功能需求

### 2.1 词法分析器修复

#### REQ-001: 中文操作符正确识别
**需求描述**:
当词法分析器读取中文字符序列时，系统应优先检查完整标识符是否为关键字或操作符，若匹配成功则识别为对应Token类型，否则识别为普通标识符。

**验收标准**:
- [ ] 中文操作符"加"被识别为 TokenType.PLUS，而非 IDENTIFIER
- [ ] 中文操作符"减"被识别为 TokenType.MINUS
- [ ] 中文操作符"乘"被识别为 TokenType.MULTIPLY
- [ ] 中文操作符"除"被识别为 TokenType.DIVIDE
- [ ] test_lexer_chinese_operators 测试通过

**优先级**: 高
**依赖**: 无

#### REQ-002: 中文标识符优先级处理
**需求描述**:
当词法分析器在 _read_chinese 方法中处理中文字符时，系统应按照"完整读取 → 关键字/操作符匹配 → 标识符回退"的顺序进行识别。

**验收标准**:
- [ ] 完整读取连续的中文字符、英文、数字、下划线组成的标识符
- [ ] 优先匹配关键字表和操作符表
- [ ] 匹配失败时正确识别为普通标识符
- [ ] 不影响现有英文标识符的识别逻辑

**优先级**: 高
**依赖**: REQ-001

### 2.2 语法分析器修复

#### REQ-003: 一元非操作符正确解析
**需求描述**:
当语法分析器遇到中文一元操作符"非"时，系统应将其解析为 UnaryOp 节点，而非函数调用。

**验收标准**:
- [ ] 表达式"非 x"被解析为 UnaryOp(op='not', operand=Identifier('x'))
- [ ] 不被错误解析为 FunctionCall(name='非', args=[Identifier('x')])
- [ ] test_parse_unary_not 测试通过
- [ ] 支持嵌套一元操作符（如"非 非 x"）

**优先级**: 高
**依赖**: 无

#### REQ-004: 多分号容错处理
**需求描述**:
当源代码中出现多个连续分号时，系统应正确处理而不引发解析错误。

**验收标准**:
- [ ] 词法分析器允许多个连续分号的词法单元
- [ ] 语法分析器正确跳过多余的分号
- [ ] test_multiple_semicolons 测试通过
- [ ] 不影响单分号的正常处理逻辑

**优先级**: 高
**依赖**: 无

### 2.3 语义分析器修复

#### REQ-005: 错误检测方法补全
**需求描述**:
系统应在 SemanticAnalyzer 类中提供 has_errors 方法，用于查询语义分析过程中是否检测到错误。

**验收标准**:
- [ ] SemanticAnalyzer 类包含 has_errors 方法
- [ ] 方法返回布尔值，True 表示存在错误，False 表示无错误
- [ ] 正确统计和报告语义错误数量
- [ ] test_undefined_variable_in_expression 测试通过

**优先级**: 高
**依赖**: 无

#### REQ-006: 未定义变量检测
**需求描述**:
当语义分析器在表达式中遇到未定义的变量引用时，系统应记录错误并通过 has_errors 方法报告。

**验收标准**:
- [ ] 正确识别表达式中的未定义变量
- [ ] 错误信息包含变量名称和位置信息
- [ ] has_errors() 返回 True
- [ ] 不影响其他语义检查的正常执行

**优先级**: 高
**依赖**: REQ-005

### 2.4 宏展开修复

#### REQ-007: 宏调用正确识别
**需求描述**:
当宏展开器处理函数调用时，系统应正确识别宏调用并执行展开操作。

**验收标准**:
- [ ] 正确判断函数名是否为已定义的宏
- [ ] 区分普通函数调用和宏调用
- [ ] 宏调用被正确展开为对应的AST结构
- [ ] 不影响普通函数调用的处理

**优先级**: 高
**依赖**: 无

#### REQ-008: 宏参数映射正确性
**需求描述**:
当宏展开器执行参数映射时，系统应按照宏定义的参数顺序正确绑定实际参数。

**验收标准**:
- [ ] 位置参数按顺序正确映射
- [ ] 参数数量与宏定义一致
- [ ] 参数类型匹配（如适用）
- [ ] 嵌套宏调用的参数正确传递

**优先级**: 高
**依赖**: REQ-007

#### REQ-009: 宏展开AST节点类型正确性
**需求描述**:
当宏展开完成后，系统应确保展开后的AST节点类型与预期一致。

**验收标准**:
- [ ] 展开后的AST节点类型正确
- [ ] 节点属性完整且正确
- [ ] AST结构符合语法规范
- [ ] 所有宏展开相关测试通过（6个测试）

**优先级**: 高
**依赖**: REQ-007, REQ-008

## 3. 非功能需求

### 3.1 性能需求
- 修复后的代码不应引入性能回退
- 词法分析和语法分析的时间复杂度保持不变
- 宏展开的深度限制应合理，避免无限递归

### 3.2 可维护性需求
- 修复代码应保持与现有代码风格一致
- 添加必要的注释说明修复逻辑
- 避免过度复杂的条件判断
- 保持代码的可读性和可测试性

### 3.3 兼容性需求
- 修复不应破坏现有功能
- 保持向后兼容性
- 测试覆盖率不应降低

## 4. 约束条件

### 4.1 技术约束
- 必须基于现有的词法分析器、语法分析器、语义分析器和宏展开器实现
- 不得引入新的外部依赖
- 修改范围限定在指定的源文件中
- 必须通过所有现有的相关测试用例

### 4.2 业务约束
- 修复优先级按测试失败影响范围排序
- 词法分析器修复优先级最高（影响后续所有阶段）
- 不得改变中文编程语言的语法规范
- 保持中文操作符的语义一致性

## 6. 术语表

| 术语 | 定义 |
|------|------|
| 词法分析器 (Lexer) | 将源代码字符流转换为Token序列的组件 |
| 语法分析器 (Parser) | 将Token序列转换为抽象语法树(AST)的组件 |
| 语义分析器 (Semantic Analyzer) | 对AST进行语义检查的组件，如类型检查、作用域分析 |
| 宏展开器 (Macro Expander) | 将宏调用展开为具体AST结构的组件 |
| Token | 词法分析产生的最小单元，包含类型和值 |
| AST | 抽象语法树，表示源代码的语法结构 |
| EARS | Easy Approach to Requirements Syntax，需求规格编写格式 |
