# EARS 格式说明

## 概述
EARS (Easy Approach to Requirements Syntax) 是一种简化的需求编写格式，旨在使需求更加清晰、可测试和可验证。

## EARS 模式

### 1. 普遍性需求 (Ubiquitous)
**格式**: `系统应 <行为>`
**适用场景**: 系统在所有情况下都必须满足的需求
**示例**: 系统应验证所有输入数据的格式

### 2. 事件驱动需求 (Event-Driven)
**格式**: `当 <触发条件> 时，系统应 <行为>`
**适用场景**: 当特定事件或条件发生时触发的需求
**示例**: 当用户点击提交按钮时，系统应验证表单数据

### 3. 状态驱动需求 (State-Driven)
**格式**: `当 <状态> 时，系统应 <行为>`
**适用场景**: 当系统处于特定状态时的需求
**示例**: 当系统处于维护模式时，系统应拒绝所有新的用户请求

### 4. 可选特性需求 (Optional Feature)
**格式**: `当 <特性> 启用时，系统应 <行为>`
**适用场景**: 针对可选功能的需求
**示例**: 当高级日志功能启用时，系统应记录详细的调试信息

### 5. 异常处理需求 (Exception Handling)
**格式**: `当 <异常条件> 时，系统应 <行为>`
**适用场景**: 处理错误或异常情况的需求
**示例**: 当数据库连接失败时，系统应返回友好的错误消息并记录日志

## 验收标准编写规则

1. **可测试性**: 每个需求必须有明确的验证方法
2. **无歧义性**: 使用精确的术语，避免模糊表述
3. **完整性**: 包含所有必要的前置条件、行为和后置条件
4. **原子性**: 每个需求应聚焦单一功能点
5. **可追溯性**: 每个需求应有唯一标识符

## 需求结构模板

```markdown
### REQ-<ID>: <需求标题>

**需求描述**:
<EARS 格式的需求陈述>

**验收标准**:
- [ ] <验收条件 1>
- [ ] <验收条件 2>
- [ ] <验收条件 3>

**优先级**: <高/中/低>
**依赖**: <相关需求 ID>
```

## 注意事项

1. 避免使用实现细节，聚焦于"是什么"而非"怎么做"
2. 使用主动语态和明确的动词
3. 确保每个需求可独立测试
4. 保持需求之间的逻辑一致性
